### C3. Turn-1 context: 13,832 vs 30,785 tokens

**Stage 0** (`stage_0:§2`): "turn 1 cache_write_1h = 13,832"
**Stage 2** (`stage_2:§7`): "Effective context (turn 1) = 30,785 tokens"

**Resolution:** Different metrics, same session.
- `cache_write_1h` (Stage 0) = new tokens entering the 1h cache on turn 1
- `effective context` (Stage 2) = total context the model sees (cache writes + cache reads + fresh input)
- The 16,953-token gap is **cache_read on turn 1** — tokens from a prior cache that are still warm
- **Action:** Both stages need a footnote clarifying these are different metrics measuring the same turn.

### SA2. S4 hole ROOT CAUSE FOUND — structurally dead, not under-sampled

**Definitive diagnosis from 194 decisions:**

1. **Max natural score = 0.400** (the highest score without guardrail clamping). S4 needs ≥ 0.550. The gap of 0.150 is unbridgeable by the signal weights.
2. **Score distribution is left-skewed:** 141/168 non-S5 scores are below 0.08 (S0). Only 2 decisions ever reached S3 naturally. None reached S4.
3. **S4-floor guardrails exist but never fire.** Keywords like "graceful degradation", "tiered memory", "state machine design", "fallback chain" don't match the user's casual communication style. Compound guardrails (design_from_scratch, full_sweep_floor, ue5_debug_floor) require_all=true and are too narrow.
4. **Every S5 decision is guardrail-clamped** — score=0.800 (from multi_file_refactor/architecture_work) or score=0.850 (from gpqa_hard_reasoning). Raw scores never reach these values naturally.
5. **Signal weights sum to 1.10** but practical ceiling is ~0.40 because most signals fire rarely and never simultaneously.

**The S4 tier is unreachable.** This is a threshold design issue, not a data gap. The "need 500+ decisions" deferral was wrong — 0/194 is definitive.

**Fix options (all require manifest edit — propose to the user):**
- A: Lower `s3_max` from 0.55 to ~0.30 and `s4_max` from 0.80 to ~0.55 — redistributes the natural score range across S3-S4
- B: Add S4-floor guardrails with the user's actual vocabulary (detected from decisions.jsonl high-signal prompts)
- C: Lower S5-floor guardrails (multi_file_refactor → min_score=0.55 instead of 0.80) — some S5s become S4s
- D: Accept S4 as dead and collapse to a 5-tier system (S0-S3 + S5)

Stage 6 reports "6 transcripts / 22MB" for the Toke project directory. Stages 1 and 3 report 25-29 sessions captured in decisions.jsonl. These are different things — decisions.jsonl captures prompts across ALL sessions in ALL projects, while transcripts are per-project. Not a contradiction but easily confused.
