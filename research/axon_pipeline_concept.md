## Key Architecture Terms (as defined)

- **Brick** — a pipeline stage/node
- **Axon** — the node itself, takes info in, processes, passes forward
- **Tubial** — the connection between axons; fires when a connection goes suboptimal
- **Author** — knowledge retrieval + passage extraction from bibles/docs
- **Self-improvement loop** — wrong-firing axon gets mapped to its own learnings position, routed to correction

## Architecture Layers (Top-Down)

1. **Scope Brick** — init, Rosetta Stone, project + scope loaded
2. **Category Brick** — task filtered into domain (movement, weapons, NPC, inventory, etc.)
3. **Author Brick** — scans relevant bibles/docs, extracts value passages for the task
4. **Tool Selection Brick** — decides which tools run (trinity, profTeam, etc.), checks for info gaps
5. **Cross-Tool Synthesis Brick** — combines tool outputs, learnings tab open, tracks WHY each tool was chosen
6. **Knowledge Compression Brick** — large field → filtered to what works with our reasoning → compiled output
7. **Output Brick** — communication, formatted to most effective format

## Error / Bug Path (Tubial System)

1. Axon fires suboptimally → Tubial fires
2. Reads the failing connection, categorizes error type
3. Author reads bug-type bibles → finds correct fix
4. Maps wrong-firing cell to learnings position
5. Saves fix pattern for future flows (self-improvement)

## Learnings Tracking

- Track WHY tools were chosen per task
- Track what made specific info "enough" to complete
- Track cross-tool methods that worked
- Route tubial patterns to domain-specific correction folders
